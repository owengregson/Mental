# Rewrite Mandate: Mental — Latency-Compensated Era Combat for Modern Paper + Folia

You are the coding agent tasked with rewriting **Mental** from scratch. Mental is a Minecraft server plugin that reproduces authentic 1.7.10 / 1.8.9 ("era") melee, knockback, and projectile combat on modern Paper (1.17.1 → 26.x) and Folia, while registering hits *faster than vanilla* through a netty-thread fast path.

This document does **not** hand you an architecture. Your **first deliverable is a new architecture that you design**, justify against the rubric below, and get approved by the owner **before you write any rewrite code**. You may lift worthy files from the current codebase, but every reuse must be re-justified against your new design — never pasted blindly. Where this document is concrete, it is concrete about the **problem**, the **constraints**, and the **domain truth** — never about the solution. The solution is yours to invent and defend.

Read every section. The invariants in §4 and the bug-classes in §5 are the hard boundary your design will be judged against.

## 1. Mission & domain context

**What Mental is.** A single-jar, zero-external-plugin-dependency combat plugin that:

- Delivers **authentic 1.7/1.8 combat feel** — knockback trajectories, combos, sprint/w-tap mechanics, hurt cadence — that are byte-measured against real vanilla 1.7.10 and 1.8.9 servers (the `legacy-lab` protocol harness) and cross-checked against decompiled server code.
- **Registers hits on the netty thread** (a "fast path"): it can pre-send the client's velocity + hurt-flinch packets before the authoritative server tick, so a high-CPS player perceives zero server-tick feedback latency, and it can **latency-compensate** knockback verticals so high-ping victims see era-correct trajectories.
- Runs across the **entire Paper 1.17.1 → 26.x range from one universal jar**, and is **first-class on Folia** (regionized threading).
- Is **server-authoritative by construction** so it coexists with movement-prediction anticheats (GrimAC, Vulcan) with no exemptions.
- Ships **16+ optional, default-OFF combat-rule modules** ported from OldCombatMechanics (OCM) — armour/damage/potion/regen/golden-apple/sword-blocking/hitbox/attack-cooldown/sweep/sounds/etc. — so a server can run standalone 1.8 combat with Mental alone, while still coexisting with OCM when both are present.

**The domain is combat *feel*.** The product is *determinism* and *era exactness*. A trajectory that is off by one block, a combo vertical that re-stamps instead of declining, a standing hit that ships 0.4 vertical instead of 0.3608 — these are visible, reputation-damaging defects. The community history (see `era-accuracy`) explicitly forbids randomized knockback and CPS-scaled mechanics: they destroyed the player bases of the servers that shipped them.

**This is a from-scratch rewrite.** The current codebase is *itself* the output of a 2026-06-04 rewrite (7 modules) that then grew organically to 16+ modules via the OCM port. It works, and its **pure-math core and measured constants are a genuine asset you must preserve**. But its *stateful orchestration and delivery machinery* has accreted band-aids that a superior architecture should make impossible. Your job is to design that superior architecture, preserve all domain correctness, and resolve every findable bug **by design** — not by porting the patches.

**The current module layout you are replacing** (for orientation only — do not treat it as a target): Gradle modules `api` (public events + facade), `common` (pure logic/SPI, no Bukkit), `core` (the plugin, compiled against the 1.17.1 floor), `compat-folia` + `compat-brigadier` (compiled against newer API, class-loaded behind capability detection), `tester` (in-server integration harness, never shipped). Package root `me.vexmc.mental`. You are free to re-partition all of this — but you must justify the partition.

## 2. Why a rewrite is warranted (problems, not cures)

These are the concrete, cited weaknesses of the **current** codebase. Each is presented as a *problem to eliminate by design*, not a thing to patch. Your architecture will be judged partly on whether these classes of problem can still occur in it.

**2.1 The knockback-delivery band-aid pile.** Knockback is applied at `PlayerVelocityEvent` (a layer that fires *twice* per hit — once from Mental's netty pre-send, once from vanilla's authoritative pass — and *later* than vanilla's own knockback delta). To make that layering "work," the current code spawns a constellation of compensating singletons, none of which are `CombatModule`s and all of which are hand-wired in `MentalPlugin.onEnable`:

- `VelocityDuplicateSuppressor` — cancels the *second* velocity packet by matching payload within `MATCH_EPSILON=0.01` inside a 2-tick wall-clock deadline. Exists only because the design emits each knock **twice by construction**.
- `AppliedTagStore` — a UUID-keyed map that smuggles per-event data from a `HIGH` apply handler to a separate `MONITOR` record handler for the *same* event, because apply and ledger-record are two priority-separated Bukkit handlers. It already had to *abandon* a 25 ms wall-clock TTL that a GC pause could strand.
- `MeleeReentryGuard` — a static `ThreadLocal<Boolean>` set around a fishing rod's `victim.damage(rodder)` call, because a rod hit's `EntityDamageByEntityEvent` is **indistinguishable by cause and damager type from a real melee swing**. Provenance is smuggled through a hidden thread flag.
- `ServerTickClock` — a plugin-maintained `AtomicInteger` surrogate for `Bukkit.getCurrentTick()`, because that call **throws off a Folia region thread** and the netty loop is never a region thread. Initialised to a `NO_TICK` sentinel so a stalled counter degrades to "no exclusion" instead of a permanent server-wide feel bug.
- `GroundPacketTap` + `GroundTransitionWatcher` — reconstruct the victim's ground/jump/liftoff state from client movement packets on the netty thread, feeding a `ConcurrentHashMap` motion ledger written by **four racing writers across three threads**.
- `PendingStore` (per-victim FIFO with `PAPER_PENDING_EXPIRY=100ms` / `FOLIA=300ms`), `SprintTracker` (**five** concurrent collections + a thread-local + a deferred-clear reconciliation dance), and `KnockbackEventMirror` (a reflective HIGH listener that mirrors the shipped value onto an event Mental doesn't authoritatively use).

Correlation between the netty pre-send and the authoritative pass is done by **heuristics** — attacker UUID + time-window + payload epsilon — never an explicit hit identity. Every seam therefore carries a **platform-forked tolerance/expiry constant** tuned to Paper's 50 ms tick that Folia's multi-region-tick gap violates.

**2.2 The Folia fix chain (≈19 fixes across 6 releases) — and the ghost it chased.** The single most fix-saturated subsystem. It began when the netty path did **live entity reads** (`getEntityById`, `getGameMode`, `getWorld().getPVP()`) that throw `ensureTickThread` off-region on Folia, were swallowed by a catch-all, and silently handed **every Folia PvP hit to vanilla** (no era model). The fixes cascaded accessor-by-accessor (`Bukkit.getCurrentTick`, cross-region attacker reads inside the synchronous `EntityDamageByEntityEvent` dispatch whose per-listener `catch(Throwable)` swallows the throw, cross-region rod casts, single-slot pending overwrites, sprint-clear timing). **The reported symptom that drove the entire 2.1.1/2.1.2 campaign was a *measurement artifact*** — the test harness read `EntityKnockbackEvent` (which Mental never writes to; Mental writes `PlayerVelocityEvent`), so ~19 fixes chased a bug that did not exist on real clients. The matrix has **zero live Folia combat coverage**, so this could only be found by hand.

**2.3 The 16-CombatModule OCM sprawl.** The OCM port more than doubled the module count (7 → 23 registered modules, ~40 `Settings` records). `MentalPlugin.registerModules()` is now a **flat, hand-wired ~25-line imperative list** behind a 60-import header, with cross-module coupling (`knockback.hints(compensation)`) wired outside the registry and shared singletons constructed in bootstrap. The `module/{rules,damage,potion,consumable,health,block,hitbox,fishing,projectile}` package taxonomy has **no consistent organizing principle** (single-module packages next to catch-all `rules/`; `sword-blocking` in `block/` but is a "rule"; ender-pearl-cooldown in `consumable/`). The OCM **coexistence** model is a *frozen 6-value enum* (`OcmMechanic`) that only understands the *original* knockback mechanics; the ~10 new rule modules **never consult the ownership gate**, so enabling the same rule in both Mental and OCM **silently double-applies** with no warning and no test — and `CritFallbackModule` already forgot the gate on one of its two code paths (crit applied twice).

**2.4 Two lifecycles, one framework.** `CombatModule` only manages Bukkit `Listener`s (auto-unregistered via `HandlerList`). **PacketEvents listeners live in a second, unmanaged lifecycle** registered directly in bootstrap for the plugin lifetime. This produces **split-brain features** (attack-cooldown, attack-sounds, sweep-particles each have a `CombatModule` half *and* a raw packet-listener half): toggling the feature in the GUI disables only the module half; the packet half keeps intercepting every matching packet and cannot be turned off without a server restart. It also produces a latent **partial-enable listener leak**: `onEnable()` registers listeners *then* throws, cleanup is guarded behind an `active` flag a failed enable never sets, and a subsequent reload registers them a *second* time (hits processed twice).

**2.5 Version-number & branding chaos.** `build.gradle.kts` hard-codes `version = "2.2.1"`, which flows to `plugin.yml`, to the public `Mental.version()` API, to bStats, and to the GUI. But git history carries **legacy StrikeSync tags `v4.0.0` / `v4.0.1` as ancestors of `main`** (`git describe` = `v2.2.0-7-...`), and the design spec states Mental **supersedes the legacy plugin v4.0.1**. A server admin upgrading "StrikeSync 4.0.1" → "Mental 2.2.1" sees a **numeric downgrade**; semver update-checkers refuse it; the API reports a false identity. The repo directory is still literally named `StrikeSync`, and `release.yml` carries bespoke logic to stop the "legacy v4.x StrikeSync tags" from out-sorting Mental's changelog. The version is a hand-typed literal disconnected from any release tag, with **no single monotonic source of truth** and no separation of build/display version from an API-compat version. (bStats is wired in despite being listed "out of scope" in the design.)

**2.6 The 2.1.0 breaking pivot left drift.** Release 2.1.0 (a) replaced the command surface with a single in-game GUI (only `/mental reload` survives for console) and (b) made knockback **global**, removing the per-player override and reshaping `setKnockbackProfile` to a global `String` setter and `KnockbackProfileChangeEvent` — **a breaking API change shipped with no `@Deprecated` transition, no `apiVersion()`, and no shim**. `docs/knockback-profiles.md` *still* documents the removed per-player API and `/mental kb set <player>` command; anyone following it writes non-compiling code. The declarative command-tree DSL + Brigadier bridge still carry their original weight for a surface that shrank to two literals, and on 1.20.6+ the command is registered **twice** (classic plugin.yml executor AND native Brigadier).

**2.7 Config sprawl and torn reads.** `MentalConfig` exposes **26 fine-grained accessors**, each doing its own `snapshot.get().x()`; hot paths call them piecemeal, so a reload can be observed **half-old/half-new within a single hit**. 16 of the 26 settings records are identical single-boolean toggles, each costing ~6 hand-synced touch points; the `Snapshot` has a **26-arg positional constructor** (an argument-order landmine). Module ids are **string literals triplicated** across `config.yml`, the parser, the GUI `Catalog`, and each module's `id()` with no compile-time link. GUI toggles **re-serialize the 331-line, 280-comment `config.yml` on every click**, stripping operator documentation on the 1.17.x floor. `config-version: 2` is written but never read; migration is a brittle structural heuristic.

**2.8 Reflection that fails silently.** ~12 modules each re-derive the same `CraftItemStack` handle-walk; the two largest files (`SwordBlockComponents` 655 lines, `WeaponAttributeTooltipHider` 550 lines) are almost pure NMS reflection dispatch that **mutates NMS item handles on the netty send thread** (outside the "netty reads only frozen snapshots" invariant — a latent NMS-lock/assertion hazard on future versions). Every reflective path degrades to a **silent no-op** on a mapping break — so a cross-version regression produces a quiet behavioral drift (a tooltip line that won't hide, a sword-block that stops working) that **no test or log detects** and the integration matrix structurally cannot see.

**2.9 Compensation on the wrong axis.** Latency compensation pre-bakes a vertical correction on a background probe cadence (every 5 ticks) into a per-player mutable slot consumed once per hit — so **only the first hit per ~250 ms interval is compensated**, every subsequent combo hit gets none (defeating the feature in its exact target scenario), the correction is stale by up to an interval, and a landing-predictor composition bug flattens combo verticals ~3 ticks / 150 ms too early. The latency probe rides the KEEPALIVE channel, whose correctness depends on out-cancelling vanilla's own mismatched-keep-alive disconnect handler — a lost or reordered cancel **times the player out**. Some of the motion primitives are **adapted from GPL-3.0 KnockbackSync into an MIT repo** — a license-incompatibility you must cleanly re-derive.

This is the core of your task. **You will not be handed a module tree, a pipeline, or an ownership model.** You will *produce* one, defend it, and get it approved.

### 3.2 The rubric — qualities a superior architecture must have

Your architecture will be judged against these properties. State, for each, *how your design achieves it*. **These are goals, not mechanisms** — invent the mechanism.

- **R1 — Single-owner side effects.** There is exactly **one** place that decides and emits a hit's velocity, exactly one that records the residual ledger, and exactly one that shapes damage. No reentry guard, no duplicate suppressor, no HIGH/MONITOR side-channel, and no "mirror the value onto the other event" bolt-on should be *necessary* — because the value is authored once and observed, never emitted-then-reconciled.
- **R2 — Provable Folia region-safety, not patched.** It must be *structurally impossible* to read a live entity, `getWorld()`, `getCurrentTick()`, or any `getHandle()`-routed accessor from the netty thread or from another region's thread. Off-thread live access should be a **compile-time or type-system impossibility**, not a hand-audited comment or a swallowing `try/catch`. Observing client movement and entity-action packets is a **DOMAIN NECESSITY** (the sole source of the victim's off-tick ground/liftoff state feeding the jump-stamp and within-tick attack-ordering model of §4.2/§4.3, and of the arrival-order wire sprint view of §4.5). What must be eliminated is the **RACY MULTI-WRITER** tap feeding a shared mutable ledger from three threads — resolved by single-writer ownership (R4/B2), **NOT** by ceasing to observe packets. No per-module `isOwnedByCurrentRegion` guard-by-omission is needed for correctness. The single global tick clock **REQUIRED** by §4.3/§4.5 must be a first-class, netty-readable **typed tick SERVICE** (init to `NO_TICK`, degrading a stalled counter to no-exclusion; boundary exclusion requires a ≤4-tick-recent sample), shared by the packet-stamp and snapshot-read sites. What R2 forbids is a raw `Bukkit.getCurrentTick()` call (throws off-region) and an ad-hoc `AtomicInteger` that can silently diverge from the real tick — **not** the tick-index concept itself.
- **R3 — Identity over timing.** The prediction (netty) pass and the authoritative (region) pass must be correlated by an **explicit hit identity**, not by wall-clock windows, payload epsilons, or attacker-UUID+time tuples. No platform-forked millisecond constant may decide whether a knock is delivered, suppressed, or adopted.
- **R4 — Single-writer state.** Each unit of per-player combat state (motion ledger, pending delivery, sprint/freshness) has **exactly one writer thread**; no per-player state is read-modify-written from two threads. *(Goal only — however you achieve single-writer ownership, e.g. confining all mutation to the owning region thread, a producer/consumer inbox, or a lock-free single-writer structure, is yours to choose and justify.)*
- **R5 — Compute-once, read-frozen.** A hit's decision inputs (immunity, OCM ownership, resistance roll, sprint+freshness verdict, latency hint, resolved profile, computed vector) are captured **once** into an immutable per-hit context and read by every later stage. Prediction and truth call **one decision function** with the same inputs; the only difference is *when* the packet ships, never *what* is computed.
- **R6 — Unified feature lifecycle.** There is **one** managed-feature abstraction whose activation is transactional and whose every resource — Bukkit listeners, PacketEvents listeners, *and* scheduled tasks — is released on any exit (clean disable OR a thrown enable); no feature lives outside the reconciler, no split-brain halves, no partial-enable leak, no phantom module whose `active()` is a lie. Teardown isolates each disposable: a throw while closing one component can never orphan the rest (no leaked tick task, no un-terminated PacketEvents). *(Goal only — a single registration scope is one candidate mechanism; choose your own.)*
- **R7 — A coherent feature taxonomy.** Features are organized by a principled domain axis (e.g., by the era mechanic they own), self-describe their presentation metadata, and are **enumerable** to build config, GUI, and coexistence — replacing the flat 23-module hand-wired switchboard and the parallel string-keyed catalogs. Adding a feature is a single-point, compiler-checked change.
- **R8 — Config that scales.** Immutable snapshot swapped atomically by reference, read **once per operation** (torn cross-section reads unrepresentable); no 26-arg positional constructor; module identity is a single typed source of truth (not quadruplicated strings); machine-written state does not re-serialize the human-documented, commented files; migration keyed on an explicit version chain.
- **R9 — Total, fail-closed coexistence.** Every era mechanic Mental can restore is a first-class ownership token; a rule effect runs **only after** a central arbiter confirms Mental owns that mechanic for that decider (per the fixed decider table of §4.11). A rule that forgets to declare its token **cannot execute**. Double-enable against a peer (OCM) is detected and warned, never silently double-applied.
- **R10 — Centralised, loud-failing compat.** One platform profile owns all version adaptation (capability detection, attribute/enchant/component resolution, NMS access). **Required** elements that fail to resolve fail *loudly at boot* (or disable the owning module with a log); **optional-since-version** elements return an explicit absence whose fallback is declared in one typed place. `null`-degrading-to-a-magic-literal must be unrepresentable. All NMS access lives behind a version-gated adapter with the remapper wired into production, not scattered string reflection.
- **R11 — Pure kernel vs orchestration seam.** The measured era-physics math (§6) is a standalone, dependency-free, unit-pinned kernel that the orchestration/delivery layer *consumes*. The boundary "pure era-physics kernel vs delivery adapter" is explicit — it is exactly the seam the current codebase blurs.
- **R12 — Stable, non-thrashing public API.** The public surface (events + facade) evolves additively with a deprecation policy, has an `apiVersion()` distinct from the display version, is discoverable via Bukkit's `ServicesManager` (not only a static holder), and is source/binary-compatibility tested. A future model change cannot silently break downstream consumers as 2.1.0 did.
- **R13 — Observability & testability of the delivered value.** There is a single seam exposing "what velocity did we ship to the wire" that both the plugin and the tests read, plus a **deterministic Folia-capable delivery test path**, so a fix is always validated against the *actual* delivered value — closing the loop that let 19 fixes land against a measurement artifact.

## 4. Non-negotiable invariants — the hard checklist (exact numbers)

The rewrite MUST honor every item below **byte-for-byte**. These bound the design; they do not prescribe it. Consult `era-accuracy`, `legacy-motion-physics`, `knockback-profiles`, `netty-fast-path`, `ocm-coexistence`, `paper-cross-version` for the full derivations and citations.

### 4.1 Era knockback math (the crown jewel — preserve verbatim)

- **Core melee knockback is byte-identical 1.7.10 ≡ 1.8.9.** The strength **parameter is IGNORED** — the base is hardcoded **0.4**. Order: halve `motX/motY/motZ` (÷2); then `motX -= dx/f3 * 0.4`, `motY += 0.4`, `motZ -= dz/f3 * 0.4`; then a **vertical cap: `if motY > 0.4 → motY = 0.4`** that clamps the **BASE ONLY** — it runs **before** bonus extras, which stack past it. **Crits add ZERO knockback.**
- **Sprint (+1 level) and the Knockback enchant are the SAME mechanism.** Each level adds **0.5 horizontal along the ATTACKER'S YAW** plus a **single FLAT 0.1 vertical** term. The vertical is **never scaled per level** (sprint+KB II = 3 levels = 1.5 h extra but still one 0.1 v).
- **Base direction is from the source ENTITY POSITION** (the attacker for melee; the **angler** for rod bobbers; the **shooter** for arrows/snowballs/eggs) — **never** projectile flight, never yaw. Only sprint/enchant *extras* are yaw-directed; only **arrow Punch** is flight-directed (**0.6/level** along flight + flat **0.1 v**). Coincident position (<`1.0e-4`) → random ±0.01/axis, finite.
- **Canonical measured wire vectors** (pin these at vector level, expectation = the engine's own output): standing **(0.4, 0.3608)** ≈ 1.994 blocks; sprint **(0.9, 0.4607)** ≈ 4.948; KB II sword **(1.4, 0.4607)**; sprint+KB II **(1.9, 0.4607)**. Standing vertical **0.3608 = −0.0784/2 + 0.4** (grounded-equilibrium seed, **never 0.4**); **0.4607 = 0.3608 + 0.1**.
- **Engine order of operations (fixed):** (1) legacy resistance roll (may cancel) → (2) direction from source position + melee range-reduction taper → (3) `base motion×friction − dir×push` with vertical ADD-or-SET and horizontal-cap-rescale / vertical-cap-clamps-**base-only** → (4) extras along attacker yaw → (5) air multipliers (airborne only) → add offsets → vertical-min floor → **SCALING** resistance → **±3.9** clamp.

### 4.2 The Minecraft motion model (identical all eras for a knocked player)

- Gravity **0.08**/tick; vertical drag **×0.98** (after gravity); horizontal air drag **×0.91**; on stone **×0.546** (= 0.6 slip × 0.91); jump impulse **0.42**; velocity packet clamp **±3.9** per axis (motion×8000 as shorts).
- Slipperiness (block under feet): stone **0.546**, ice/packed-ice **0.8918**, slime **0.728**, blue ice **0.989**, default **0.546**.
- **Integration order every tick:** (1) input accel → (2) **friction factor chosen from the PRE-move ground state** → (3) move by motion (collision lands entity; vertical collision zeroes motY) → (4) `motY = (motY − 0.08)×0.98`, horizontal ×the step-2 factor. **A knock's launch tick decays at GROUND drag even though the move lifts the victim.** Two grounded pre-move decays precede the air segment: residual = `0.4 × (slip·0.91)_knock × (slip·0.91)_liftoff × 0.91^k`. (Post-move friction overshoots a 0.4 knock by a full block — the #1 trap.)
- **onGround refresh only while moving downward.** Zeroing motY on a standing entity mis-reads it as airborne (next launch takes AIR friction — a silent block-scale error). Any victim-motion reset clears **horizontal axes only** and preserves the **−0.0784** vertical equilibrium.
- **Jump stamp:** at any rising liftoff (`onGround && !packet.onGround && dy>0`) the handler runs `jump()`, **overwriting motY = 0.42** + `0.1×(amp+1)` if Jump Boost + a **0.2** facing push if sprinting. Consequence: **combo verticals DECLINE** (1.8.9 combo hit 2 ships vy ≈ 0.2477–0.2862, stamp-phase dependent); victims stay low. Jump-Boost amplitude must be cached per player.

### 4.3 The residual ledger & combos

- **Send-then-restore:** melee knockback to a **player** victim is a *packet*; the server's own motion copy is restored immediately (1.8.9+ and still today). **`getVelocity()` is stale/reverted for players on every supported version.** Mental keeps the legacy fields itself in a per-victim residual ledger — recording the **FINAL delivered value** (what the client actually received after every third-party velocity handler ran), bound to the hit-identity as **part of the single apply-and-record operation** (see B4 — **not** a separate `MONITOR` handler sharing a UUID-keyed side-channel), decayed by the legacy friction model, **dead after 60 ticks**. A pinned hit cancelled downstream is never recorded; a pre-delivered (wire-carried) hit is recorded as shipped. **Mobs are server-simulated — their live velocity is the correct input.** The engine must **never read `getVelocity()` for players**.
- **Combo era switch:** `combos:true` = **1.7** (applied knocks persist, melee recorded, both join orders compound); `combos:false` = **1.8** (melee applies **not** recorded — send-then-revert kills melee combos — but **rod/arrow/thrown residuals STILL record** in both modes). **Default is 1.7 (combos on).**
- **Within-tick attack ordering:** the attack is judged **BEFORE** the victim's same-tick movement. A combo hit thrown the instant its victim touches down reads the **pre-landing flight** (ships the declining ~0.25 vertical), **never** a grounded 0.3608 re-stamp. Replicate via: (a) read the residual as of the **END of the previous tick** (`currentExcludingTick`, with a recency guard), (b) a **+1 staleness allowance** in the invuln check so boundary hits at `max/2 + 1` are admitted, (c) the `auto` feedback window at **`max/2 − 1`** ticks. A **single global tick clock** feeds both the packet-stamp and snapshot-read sites, and **initialises to `NO_TICK` (not 0)** so a stalled counter degrades to no-exclusion; the boundary exclusion requires a **recent (≤4 tick)** sample.
- **Attacker-side ×0.6 writes the SERVER's attacker fields:** `attack()` runs `this.motX *= 0.6; this.motZ *= 0.6` on the server's copy of the attacker's motion on every bonus-knockback hit (beside the sprint clear, both eras). Apply as `ledger.scaleHorizontal` on accepted bonus-KB melee hits (owning-thread). This is **distinct from** the untouchable client-side 0.6.
- **The FULL stamp is era truth and the default for every preset.** The decayed wire (the later-joiner bimodal mode, standing ≈0.99 / sprint ≈2.54 — the "relog for less KB" folklore) is an **explicit opt-in, never a default;** the opt-in decayed vectors are standing **(0.2184, 0.2751)** ≈ 0.99 blocks and sprint **(0.4914, 0.3731)** ≈ 2.54 blocks (one decay tick less than the full stamp). A "players only move ~1 block" report means the decayed wire is shipping — a bug/misconfig, **not** era truth (this verdict was reversed once already). Projectile delivery is **tracker everywhere** (rod/arrow rode the tracker both eras).

### 4.4 The untouchable client-side technique contract

Preserved **by construction**, **never** reimplemented server-side: the **0.6^clicks attacker self-velocity multiplier** (why high CPS reduces received KB); **W-tap / S-tap** sprint re-arming; **jump-resets**; **A/D strafe** redirection. A real victim's walk/jump input is integrated **by the client on top of** the knock packet and **never enters server motion fields or the residual ledger**. The server sprint flag stays **client-authoritative**.

**The one exception — block-hit sprint reset.** The *sole* technique needing server-side reconstruction on modern clients: 1.7/1.8 block-start sent STOP_SPRINTING and release re-armed the sprint-KB bonus, but modern (1.17+) clients **keep the sprint flag through item-use**, so that STOP/START never crosses the wire. Reconstruct it on the block right-click, **gated on the RAW client sprint flag** (so a stationary defensive block gains no phantom bonus), and do **not** attempt to remove the client-authoritative sprint-particle visual.

### 4.5 Delivery, netty, and Folia threading

- **Delivery arbitration (last-submitter-wins).** All knockback sources — melee, rod, projectile, arrow, other plugins' velocities — submit a computed vector to **ONE** pipeline; the **LAST submission for a victim within a tick WINS** (one hit → one knock). This is how a fishing-rod hit's vector correctly supersedes the melee vector its own `victim.damage(rodder)` call provokes in the same tick, and how any later legitimate source overrides an earlier one. It is **NOT** first-wins, **NOT** priority-wins, and **NOT** a merge. Modules are pure vector computers; they never apply or record directly — the single pipeline owns **ALL** velocity application and ledger recording, with the fixed engine order of §4.1. *(Distinct from the 20-tick invuln window in §4.6 by which a rod suppresses a later melee hit — that is a different mechanism.)*
- **Netty thread reads ONLY frozen snapshots** — no live entity, no `getWorld().getEntities/getNearbyEntities`, no `getEntityId/getName/getGameMode`, no `Bukkit.getCurrentTick`, no `SpigotConversionUtil.getEntityById` on cache miss. **SAFE cached reads:** `getLocation`, `getWorld`, `getUniqueId`, `getType`, `isValid`, `isDead`, `isOnGround`, `world.getPVP`, `Bukkit.getPlayer(uuid)`, `Bukkit.getOnlinePlayers`. Resolve player victims via a **frozen entityId→UUID index** + `Bukkit.getPlayer(uuid)`; non-player targets unresolvable off-region **pass through to vanilla on Folia**.
- **Two knockback events.** `EntityKnockbackEvent` (Paper 1.20.6+) fires *inside* the damage pass with vanilla's **DELTA** (airborne vertical delta is 0). `PlayerVelocityEvent` fires *later* in `ServerEntity.sendChanges` just before the motion packet — this is the **ONLY** event Mental authoritatively overrides, so real clients get Mental's value on Paper AND Folia. Anything reading `EntityKnockbackEvent` sees **vanilla's** value. **Apply in exactly one place** (the late velocity event); on 1.20.6+ **non-destructively mirror** the early event so mid-pass observers read the same delta, sharing the win/adjusted state with the apply so they can never diverge.
- **Pre-send composition (pure, unit-pinned):** velocity packet **BEFORE** hurt, always. On 1.19.4+ wrap in **bundle delimiters** (one write each + single flush); **never** a single-packet bundle; below 1.19.4 bare back-to-back. Use **HURT_ANIMATION** (explicit yaw), **NEVER DAMAGE_EVENT** (clients couple damage-type effects to it and the authoritative re-send double-fires); entity-status **2** fallback below 1.19.4.
- **`submitPinned` vs `submitPreDelivered`:** a victim with **no PacketEvents user** (in-process bots, synthetic/disconnecting players) gets **no burst** and must **never** be accounted wire-delivered — it ships once through the normal velocity event, **no suppressor armed**. `submitPreDelivered` means the wire **carried** it (suppress the authoritative duplicate). Conflating them drops knockback for any non-PE victim whenever a downstream suppressor fires. `getUser(player)` null → guard and **pin**. Velocity suppressors (hurt still ships): anticheat gate, OCM owning attacker KB, pending LEGACY resistance roll, missing snapshots, the feedback window.
- **Cancelling the ATTACK packet leaves vanilla obligations** that must be re-implemented: (1) `setSprinting(false)` after every accepted sprint-bonus hit (the server half of w-tap — measured: without it, no-w-tap and w-tap both fly 10.09 blocks; real 1.8.9 separates them **7.2 vs 11.4**); (2) an **attack-time sprint+freshness verdict stamped at registration** and consumed by the authoritative pass (never a live re-read); (3) an arrival-order **wire sprint view** fed by entity-action packets (Bukkit toggle events must **NEVER** write it). **Deliberate omissions stay omitted:** sweep, durability, statistics, hunger.
- **Folia region-safety:** all entity work via a region-correct scheduler (`repeatOn(entity,…)`); **no `BukkitScheduler`, no global entity-iterating tick.** The authoritative path reads the live **attacker** on the **victim's** region thread; reading a cross-region attacker throws `ensureTickThread`, and inside the synchronous `EntityDamageByEntityEvent` dispatch the event bus **swallows that throw** — leaving vanilla to apply a raw knock over a half-mutated ledger. **INVARIANT:** no attacker live-read may occur off its owning region, and a cross-region melee hit is **DROPPED (logged skip)** — the only region-safe outcome, intended behavior, not a bug to "fix." *How* you guarantee this is your design's job (R2: make it structurally impossible, not a per-handler `isOwnedByCurrentRegion` guard-by-omission). Debug suppliers on netty must use cached names, never `player.getName()`.
- **The shared tick clock** initialises to `NO_TICK`, both sites read the one clock, boundary exclusion requires a recent (≤4 tick) sample.

### 4.6 Damage, blocking, invulnerability (the legacy-damage module, default-OFF, but exact)

- Sharpness **1.25 × level** (1.9 changed to 0.5×level+0.5). Crit **×1.5 on weapon damage BEFORE enchantments**, **no sprint exclusion** (the exclusion is a 1.9 rule). Pre-1.9 tool tables: **sword 4+tier (diamond sword = 8)**, axe 3+tier, pickaxe 2+tier, shovel 1+tier. **Reset the vanilla attack-charge meter per hit** so 1.9 scaling never applies. Weapon damage keys off the **effective material** (§4.9). When Mental hands OCM-shaped damage, it hands **vanilla-shaped** (attribute base, 1.9 crit/sharpness), **not** legacy-composed, and never resets the charge (era pin: sharpness-5 diamond = **14.25 not 17.5**).
- **Blocking halves damage AFTER knockback already ran**, so era blocked hits knock **FULL** — cancel only **FULL** blocks. A stronger hit mid-invuln (`hurtResistantTime > max/2`) deals **difference damage (`amount − lastDamage`) with NO knockback, no hurt animation, no sound**; a weaker-or-equal mid-invuln hit does nothing. `maxHurtResistantTime = 20` both eras (effective 10-tick cadence). **Any accepted 0-damage hit knocks FULL and arms the full 20-tick window.** Pure-vanilla snowballs/eggs/rod bobbers **never knock player victims** (the `amount==0` gate) — era rodding is CraftBukkit-lineage; rod/projectile knockback is supplied as opt-in **as-played** behavior (from source position, arming the 20-tick window at `lastDamage=0`, so a rod suppresses a melee hit inside 10 ticks).
- Armour resistance is **probabilistic all-or-nothing**; era pools couldn't trigger the roll → **default none**. Fork presets ship **armor-resistance: none**.

### 4.7 Zero-touch, era-exact defaults, atomic config

- **Zero-touch:** a disabled feature does **nothing** observable — no velocity write, no packet, no damage change, no event mutation. Always-on infrastructure may **observe** but **never act** while its owning feature is off, and must be structurally incapable of writing state.
- **Era-exact no-op defaults:** every new knob defaults to byte-identical legacy behavior; **`parse(empty) == LEGACY_17`**, test-enforced, plus a per-knob no-op-at-default property. Adding a knob may never shift the default feel.
- **Atomic config:** one immutable `Snapshot` swapped by reference — no code path may read a torn mix mid-hit. Warn-and-fallback parsing (absent → silent fallback; wrong-type → one named warning + fallback). Per-concern file split. Automatic v1→split migration (backup + tuned knockback → `profiles/custom.yml`, selected). Netty reads the profile **frozen into the per-tick snapshot** so prediction and truth share one profile.

### 4.8 Presets & profiles

- A profile is the **whole engine knob set** in one file under `profiles/<name>.yml`. **Ten presets:** legacy-1.7 (default = era-exact no-op), legacy-1.8, kohi, minehq, badlion, velt, mmc, lunar, signature, custom. **Friction knobs are surviving-fraction multipliers** (vanilla ÷2 ≡ 0.5, NachoSpigot convention); a fork that publishes **divisors** ports as **1/d** (pasting a divisor unchanged **inverts** the feel — the #1 preset error). Preset values come from **real archived server configs** (two independent archives, byte-identical where they overlap) with **cited provenance** — not remakes.
- Presets are extracted **only when missing**; **owner edits are sacred** (deleting regenerates pristine; **any tuned value freezes the file forever**); the **one exception** is a file still matching a superseded bundled revision **by value-equality**, upgraded in place when research corrects a preset. **Knockback is GLOBAL** (since 2.1.0): resolved by the victim's **per-world map → server default**; **no per-player override**, no `/mental kb set <player>`, no per-player API/event.

### 4.9 The `combat:effective_material` cross-plugin contract (public interop — preserve)

A vendor-neutral, read-only contract: a display-swapping plugin stamps `NamespacedKey("combat","effective_material")` (`PersistentDataType.STRING`, an exact `Material` enum name) on an item whose display material is weaker than its true material; era-combat plugins **read** it to apply the true material's legacy stats. **Absent → use `getType()`** (pure no-op for unmarked items); **unknown name → degrade to the type, never throw.** Honour it **only where the relevant restore module is enabled** (Mental owns the era decision; the marker never forces legacy values on). Weapon legacy damage + tooltip re-valuation read the effective material; **armour** works via the summed `GENERIC_ARMOR` attribute; **durability** breaks against the `max_damage` component (1.20.5+). Reference writer: StarEnchants heroic upgrade.

### 4.10 Cross-version, PacketEvents, and anti-features

- **Floor-API compile (1.17.1)** so the common path is binary-safe everywhere; newer-API features behind runtime **capability class-presence** detection (never version-string branching for behavior). `release = 17` bytecode; ≤1.20.4 Java 17, 1.20.5+ Java 21+; one universal jar, `api-version 1.17`. Known breaks absorbed by boot-time resolvers: 1.21.3 Attribute enum→registry (resolve by name, modern-first-then-legacy); 1.20.5 Mojang-mapping flip (route reflection through a remapper, parse reobf mappings **once per JVM**); 1.21.2 join-protection relocation (three layouts) and vanilla projectile-KB-vs-players restored (**projectile substitution becomes a no-op there**); 1.19.4 HURT_ANIMATION/DAMAGE_EVENT/bundles appear. **Mechanics MOVE between classes across versions** — disassemble the real server (`javap` on Mojang-mapped jars, see `nms-archaeology`) to confirm a relocated mechanic, never guess.
- **PacketEvents discipline:** shaded + relocated (`me.vexmc.mental.lib.packetevents.*`); built in `onLoad`, `init()` **after** module listeners register, `terminate()` **after** they unregister; `getUser(player)` is **null** for synthetic/disconnecting players (guard and skip); listeners fire for **EVERY** connection state — **never downcast `getPacketType()` to a `PacketType.Play.*` family** (a real client's join traffic throws CCE per packet — shipped once in 1.3.0); **compare by reference** against specific constants; note Configuration has its own KEEP_ALIVE/PONG that must not match a Play guard; wrap sends to possibly-reconfiguring targets (1.20.2+) in catch-all and drop. Pin the pre-Play connection-state contract in **unit tests** against synthetic events (the live matrix structurally cannot see it — fake players join directly in Play state).
- **Latency-probe transport:** use a channel with **no disconnect semantics** — the Play **PING/PONG** channel (present across the whole 1.17+ range), behind a `Probe` abstraction. The **KEEPALIVE** strategy (whose correctness hinges on out-cancelling the server's own mismatched-keep-alive disconnect handler — a lost or reordered cancel times the player out) must be **dropped**, or hard-gated behind an explicit unsafe/legacy flag with a loud startup warning.
- **Reach validation (P5) is default-OFF**, a ping-rewound blatant-reach *filter* (not an anticheat): 40-sample/tick position ring, **biased to allow** (reject only when **every** rewound candidate exceeds range + leniency), defers to detected anticheats, skips creative/untracked parties. Evaluate reach against a **single reconstructed instant for both parties** (attacker eye and victim box at the same rewound moment).
- **Anti-features the community history forbids — do NOT implement, and do NOT "fix":** randomized knockback (determinism IS the product); CPS-scaled anything / "anti-reduce" normalization (fairness features must be explicit opt-ins, never defaults); hit-delay / `noDamageTicks` knobs (belong to OCM). **Debunked myths** to not implement: victim sprint-key-spam reducing knockback (placebo); "1.7 animations reduce KB" (placebo — the real incident was a patched 1.12 Lunar client velocity bug).

### 4.11 OCM coexistence — deciders, handoff & startup warnings

- **Per-mechanic ownership decider (fixed — the R9/B14 arbiter keys on this):** **melee knockback, tool-damage, and crits → the ATTACKER's modeset**; **fishing → the RODDER's**; **thrown projectiles (snowball/egg) → the VICTIM's**; **arrows → ALWAYS Mental**. A rewrite that resolves melee-KB ownership by the victim instead of the attacker silently double-knocks.
- **BOUND vs CONFIG resolution.** `OcmGate.handles(mechanic, decider)` in **BOUND** mode calls OCM's reflective service API on the **OWNING thread ONLY** and **FREEZES** the per-interaction verdict into the victim's per-tick snapshot for netty reads (never call OCM from netty); in **CONFIG** mode it falls back to conservative **GLOBAL** verdicts (yield rather than double-apply — a double-knock is worse than a deferred one).
- **Vanilla-shaped handoff.** When OCM shapes damage, Mental hands the hurt pipeline **VANILLA-shaped** damage (attribute base, 1.9 crit/sharpness), **never legacy-composed**, and **never resets the charge** (lossless round-trip; era pin: sharpness-5 diamond = **14.25, not 17.5**).
- **Two silent feel-burying OCM defaults — WARN at startup.** `old-player-knockback` is active in OCM's default `old` modeset (so OCM then owns melee KB and buries Mental's feel), and attack-frequency `playerDelay:18` ≠ era **20**. These are the **FIRST** things to check on any "Mental knockback is broken" report — Mental must warn on both at startup.
- **Mental's own ported rule modules are OCM-agnostic:** enabling the same rule in both plugins **double-applies** — pick one per rule. The arbiter above governs only the mechanics OCM historically owns; the ~10 new rule modules must document their double-apply risk, not silently double it.

## 6. Reuse policy

The current codebase's **pure-math core and its hand-pinned unit tests are the measured-on-real-hardware oracle** and the highest-value asset. **Lift the math and its tests near-verbatim; redesign the stateful glue around them.** Any reused file must be **re-justified against your new architecture** and its unit pins carried over unchanged — a rewrite that cannot reproduce these exact constants has regressed era accuracy.

**Lift near-verbatim (math + tests):**

Your architecture should make staleness **structurally impossible** (embed a per-run nonce / signal via exit code), not rely on the manual "check the mtimes" ritual.

6. **OCM coexistence is CI-gated** (acquire the OCM artifact reproducibly, not from a manually-copied out-of-repo jar) and covers the ported RULE modules for double-apply, not just the original 6.

8. **Single machine-readable support-matrix descriptor.** The supported-version set and its per-version JDK / platform / suite mapping live in **ONE** machine-readable descriptor read by every consumer (Gradle task registration, both CI workflows, the local concurrent runner, the release-notes range string, `plugin.yml` `api-version`). **No Minecraft-version literal or JDK-boundary value may appear in more than one place** (the current tree already diverges — Gradle maps 1.20.0–1.20.4 to Java 17 while the Bash runner maps it to Java 25); both the PR gate and the release gate derive their matrix from it, and **Folia is a first-class entry with live combat coverage**.

**Definition of done (the rewrite):** the plugin ships one universal jar over 1.17.1 → 26.x + Folia; all of the above passes on fresh results; every §4 invariant holds byte-for-byte; every §5 bug-class is resolved by design; the reused math pins pass unchanged; the version/branding chaos is resolved (§9); and the docs are reconciled to the shipped model.

## 9. Working method & phasing

**Design-first. Do not write rewrite code until the architecture is approved.**

1. **Explore & confirm** the current behavior and the reusable assets (use `explore` / `feature-research`). Read the skills and docs in §7.
2. **Brainstorm ≥ 3 candidate architectures** (`superpowers:brainstorming`), evaluate against the §3.2 rubric and the §4 invariants, and **write an architecture/decision document** presenting the options, trade-offs, and a recommendation. **Present it to the owner for approval.** This document is the **first deliverable** and its acceptance criterion is §10's closing definition of done. The approval is the hard human checkpoint of §3.1 — do not self-approve, do not proceed on a standing autonomous-run grant.
3. On approval: **write the spec** (`superpowers:writing-plans` after a spec), then a **phased implementation plan** — pure-math kernel first (port the pins), then the single-owner delivery core, then the feature/lifecycle framework, then features, then compat, then GUI/config, then tester. **TDD each subsystem** (`superpowers:test-driven-development`): failing test → implementation → verify.
4. **Commit as you go** with **conventional commits and prose bodies** explaining the *why*. Branch first if on the default branch. Follow the commit trailer convention.
5. **Resolve the version/branding chaos deliberately** as an explicit task: pick a single monotonic version scheme with **one source of truth** (derive from the release tag; keep it **≥ the superseded 4.0.1** so upgraders never see a downgrade; add an `apiVersion()` distinct from the display version); clean `plugin.yml` and branding; shed the StrikeSync cruft (the repo directory still named `StrikeSync`, the legacy `v4.x` tags in changelog logic, the bStats-present-despite-"out-of-scope" mismatch). Decide GUI-vs-commands and the compat-module partition **intentionally**, not by inheriting vestigial architecture.
6. **Reconcile the docs** to the shipped model as part of done: delete the per-player-override narrative from `docs/knockback-profiles.md`; align the profile-resolution docs and any command surface with the **global** model; generate/verify the profile docs against the code so they cannot drift again.

Follow all conventions in `mental-conventions`: immutable records + atomic config snapshots; pure math with hand-computed unit pins; netty threads read only frozen snapshots; entity work via the region-correct scheduler; imports never inline-qualified; comments explain the *why*.
