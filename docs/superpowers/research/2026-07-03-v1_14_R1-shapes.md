## Headline: zero FakePlayer code changes

The tester's legacy `FakePlayer` is **100% reflection-scan-driven** — there is no
hardcoded `v1_..` switch or list anywhere in `tester/src` (the only `v1_16_R3`
mentions are doc-comments). Every legacy branch is a self-adapting reflective
scan, and **every one resolves correctly for `v1_14_R1`**. So no code change was
required to support 1.14.4; the "impossible hole" was purely the terminal build's
Java-13 runtime cap, never a missing FakePlayer shape — and the mega-jar's v52
base tree classloads on the Java 8–13 JVMs 1.14.4 accepts, closing it.

**1.14.4 straddles the eras.** It carries the 1.15-era `Entity`/motion/
interact-manager/`DedicatedServer` shapes **but the 1.13-era synchronous join** —
Paper's async login split landed in **1.15, NOT 1.14**. The existing
`legacyAsyncJoin()` probe (marker: a private `postChunkLoadJoin` on `PlayerList`)
routes it correctly regardless; the only artifact was one inline comment that
said "Paper 1.14+ … split login" — factually 1.15+, now fixed in `FakePlayer`.

Detail on the load-bearing rows:

- **Row 7 (the key finding).** A whole-class scan of `v1_14_R1.PlayerList` for
  `getChunkAtAsynchronously` / `CompletableFuture` / `thenAccept` /
  `postChunkLoadJoin` returned **empty**. `PlayerList.a` disassembly shows it
  fires `PlayerJoinEvent.<init>` inline (offset 912), calls
  `WorldServer.getChunkAt(int,int)` synchronously (offset 249), and
  `WorldServer.addPlayerJoin(EntityPlayer)` inline (offset 1201) — no future, no
  callback. `legacyAsyncJoin()` therefore returns **false**, and FakePlayer takes
  the synchronous `PlayerList.a` branch, exactly as on 1.9.4–1.13.2.
- **Rows 2 & 3 (the modern straddle).** `PlayerInteractManager(WorldServer)` and
  `Vec3D mot`/`setMot` both **begin at 1.14.4**, not 1.15.2. The
  `2026-07-02-legacy-fakeplayer-nms-shapes.md` divergence rows placed these at
  1.15.2 (that doc only sampled the viable-tier set, which skipped 1.14.4); they
  are corrected in place there, dated 2026-07-03. The reflective scans handle it:
  `setMotionLegacy` tries `setMot(ddd)` first, and PIM resolves by
  `isAssignableFrom(worldServer.getClass())`.

## Loader / PDC / AbstractArrow / api-version oddities

- **api-version.** `CraftMagicNumbers.SUPPORTED_API` is built in `<clinit>` as
  `Arrays.asList("1.13","1.14")`. `checkSupported` does `indexOf(apiVersion)`:
  a missing api-version logs the "does not specify an api-version" WARNING and
  loads as legacy; an unknown one (e.g. `1.16`/`1.17`/`26.1.2`) is **hard-rejected**
  with "Unsupported API version" — stricter than pre-1.13 servers, which have no
  api-version gate at all. Mental's built `plugin.yml` declares
  `api-version: '1.13'` (index 0, the floor) → accepted cleanly on 1.14.4.
- **PersistentDataContainer IS present** — 1.14 is the **first** version with PDC.
  `Entity extends PersistentDataHolder`, `getPersistentDataContainer()` is
  declared, and `org/bukkit/persistence/PersistentDataContainer.class` is bundled.
  So Mental's PDC probe resolves **PRESENT** on 1.14.4 — the pre-PDC metadata/NBT
  fallback used on 1.9.4–1.13.2 must NOT engage here. 1.14.4 is the boundary.
- **AbstractArrow IS present** (added in 1.14): `interface AbstractArrow extends
  Projectile`, `interface Arrow extends AbstractArrow`. So the projectile-KB
  handlers that name `AbstractArrow` resolve on 1.14.4 but are dormant on
  1.9.4–1.13.2 — a 1.14+ divergence a probe must gate on, never "pre-1.17 == no
  AbstractArrow".
- **Plugin-loader multi-release awareness (round-2 map).** The legacy Bukkit
  `PluginClassLoader` opens its jar **plain** (`new JarFile(File)`) on
  1.12.2/1.13.2 and reflectively-versioned from 1.16.5 (Java 9+), directly
  versioned on 1.17.1, and Paper's `FileProviderSource` on 26.x opens the
  classloading jar versioned for both plugin pipelines. **1.14.4 is moot** for
  this axis: its terminal build hard-caps at Java 13 (< 17), so `versions/17` is
  out of reach for its runtime whatever the loader does — it reads the **base
  tree (major 52)**, live-asserted by the tester's `bytecodeTier: 52`.

## The straddle, in one line

1.14.4 pairs 1.15-era NMS internals (`Vec3D mot`, `PlayerInteractManager(WorldServer)`,
`DedicatedServer`/`DedicatedPlayerList`, PDC, AbstractArrow) with a 1.13-era
synchronous player-join — but critically it runs **only under Java 8–12** (Spigot
1.14.4's Java-13 hard cap), which is why it stayed outside the Java-17 matrix
regardless of these shapes lining up, until the mega-jar's v52 base tree made it
loadable on those JVMs.
